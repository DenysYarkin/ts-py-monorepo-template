This skill is for database schema changes in `apps/api` (FastAPI + SQLAlchemy + Alembic).

## Repository Wiring

Migration behavior is defined by these files:

- `apps/api/alembic/env.py`
  - Imports `DB_URL` and `Base` from `database.db`.
  - Imports `database.models` to register all models in `Base.metadata`.
  - Overrides `sqlalchemy.url` with `DB_URL` at runtime (`config.set_main_option(...)`).
- `apps/api/database/db.py`
  - Loads `.env` with `load_dotenv()`.
  - Builds `DB_URL` from `DB_USERNAME`, `DB_PASSWORD`, `DB_HOST`, `DB_PORT`, `DB_NAME`.
- `apps/api/database/models/__init__.py`
  - Must import every model module/class that should be visible to Alembic autogenerate.
- `apps/api/alembic.ini`
  - Configures script location and filename template for migrations.
  - Uses timestamp-based naming: `YYYY_MM_DD_HHMM-<revision>_<slug>.py`.
- `apps/api/alembic/versions/`
  - Stores migration files.

## Commands (run from repository root)

- Create empty migration:
  - `npx turbo run migrations:create -- -m "<short sentence>"`
- Create autogenerated migration:
  - `npx turbo run migrations:create:auto -- -m "<short sentence>"`
- Apply pending migrations:
  - `npx turbo run migrations:run`
- Revert last migration:
  - `npx turbo run migrations:revert:last`

These commands map to Alembic scripts in `apps/api/package.json`.

## Standard Workflow

1. Change SQLAlchemy models in `apps/api/database/models/`.
2. Ensure changed/new models are imported in `apps/api/database/models/__init__.py`.
3. Generate migration with a concise sentence message.
4. Review migration file in `apps/api/alembic/versions/`:
   - `upgrade()` and `downgrade()` are both valid and intentional.
   - Autogenerated operations are adjusted where needed.
5. Apply migrations.
6. If needed, rollback exactly one revision.

## Quality Rules

- Do not trust `--autogenerate` blindly. Review and correct output before considering it done.
- Prefer reversible and deterministic migrations.
- For destructive changes (drop/rename/type conversion), use a safe staged approach when data loss is possible.
- Keep migration naming consistent with project convention: short sentence message.
- Do not rewrite existing committed migration history unless explicitly requested.

## Common Issues and Root Causes

- Autogenerate misses expected table/column changes:
  - Root cause is usually missing imports in `apps/api/database/models/__init__.py`.
- Migration points to unexpected database:
  - Root cause is incorrect `.env` DB vars; Alembic uses `DB_URL` from `database/db.py`.
- Alembic CLI unavailable:
  - Root cause is Python environment/dependencies not available in current shell.
- Migration works locally but breaks in shared environments:
  - Root cause is assumptions about existing data/state; make migration logic explicit and robust.

## Investigation Checklist for Agents

When asked to investigate migrations in this repo, inspect this chain in order:

1. `apps/api/package.json` for migration scripts.
2. `turbo.json` for task names and execution setup.
3. `apps/api/alembic/env.py` for DB URL and metadata wiring.
4. `apps/api/database/models/__init__.py` for model registration.
5. `apps/api/alembic/versions/` for current migration style and history.
